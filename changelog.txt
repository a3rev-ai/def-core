*** Digital Employee Framework - Core Changelog ***

= 2.4.0 - 2026/04/25 =
* Feature: Customer Chat add-to-cart now calls WooCommerce's Store API (`/wp-json/wc/store/v1/cart/add-item`) directly from the widget instead of the custom `tools/wc/add-to-cart` REST endpoint. Store API is the same endpoint the WC Cart Block uses — it manages guest sessions via a `Cart-Token` response/request header pair (separate from the `wp_woocommerce_session` cookie that edge proxies routinely strip on `/wp-json/*`), and uses its own `wc_store_api` Nonce. The browser tracks the rotated `Cart-Token` in `localStorage` so the cart survives reloads and proxy churn. Logged-in and guest visitors both flow through the same code path.
* Removed: legacy `wc_add_to_cart` PHP endpoint (~340 LOC), `permission_check_add_to_cart`, `parse_wc_session_cookie_from_request`, and the `/tools/wc/add-to-cart` route registration. Browser-side dispatch via `handleWpRestCall` is unchanged in shape — only the URL and headers it sends differ. The new path eliminates the REST-context session-init timing bug that produced "second add wipes cart" symptoms for guests under any cookie-stripping proxy (Cloudflare, hosting CDNs).
* Compatibility: `def-core-cart-sync.js` (iframe integration helper) is still enqueued whenever WC is active — its gating no longer references the removed endpoint. No JS-side breaking changes for iframe deployments.

= 2.3.2 - 2026/04/24 =
* Fix: Customer Chat tool_done status mapping is now centralised at the widget level. The widget's `completeToolStatus()` now treats `'success'`, `'completed'`, `'ok'`, and `'done'` (case-insensitive) as success, plus an empty/missing status for legacy backwards-compat. Kills the bug class behind both `Done (failed)` (DEF PR #202 fix #3 patched per-tool by translating `"completed"` → `"success"` for spawn_sub_agent only) and `File analyzed (failed)` (file extractor was emitting `"completed"` and being rendered red). Companion to DEF #204 follow-up.

= 2.3.1 - 2026/04/23 =
* Fix: Customer Chat add-to-cart now narrates from WooCommerce's actual response instead of a hardcoded "Product added to cart" string. `wc_add_to_cart` clears WC notices before dispatch, captures `wc_get_notices('error')` if `WC()->cart->add_to_cart()` returns false (e.g. `Sorry, you cannot add another "X" to your cart.`, `... not enough stock — N remaining`, `... we have N in stock and you already have M in your cart.`), and otherwise generates the success message via `wc_add_to_cart_message($return=true)`. The trailing `<a class="wc-forward">View cart</a>` anchor that WC bolts onto several error messages is stripped before `wp_strip_all_tags()` so the final string reads as natural English instead of `"…to your cart. View cart"`.
* Fix: When `handleWpRestCall` gets `{error: true, message: …}` back from a `wp_rest_call` endpoint, the JS now appends WooCommerce's actual error wording as a real assistant chat bubble (in addition to the existing red toast). Without this, the LLM's pre-execution "I'm adding it" narration was the only visible message — the user had no in-conversation evidence of WC's rejection. The new bubble surfaces the truth even when the LLM commits to a queue-language response before the wp_rest_call resolves.
* Fix: When the request was for a variation (`variation_id > 0`), the success message now keys off `$variation_id` (not the parent `$product_id`) so the chat narration uses the specific variant's display name — e.g. *"WP Email Template — Pro"* — instead of the parent product name. Matches the wording the customer sees in WC's own notice area on a normal page submit.

= 2.2.10 - 2026/04/22 =
* Fix: Customer Chat add-to-cart no longer wipes existing items from a logged-in user's cart. v2.2.6 (#137) stopped sending `X-WP-Nonce` on `wp_rest_call` actions to fix anonymous-visitor 403s, but WordPress REST cookie auth refuses to set the current user from the auth cookie without a matching nonce — so logged-in users were treated as anonymous, `wc_add_to_cart` saw `get_current_user_id() = 0`, skipped its persistent-cart merge branch, and `WC()->cart->cart_contents` started empty. The subsequent `add_to_cart()` then wrote a single-item cart and `calculate_totals` → `set_session()` overwrote the existing session cart. `handleWpRestCall` now sends `X-WP-Nonce` when `config.isLoggedIn` is true (in addition to the existing `auth: true` opt-in). Anonymous visitors still skip the nonce — preserves the v2.2.6 fix.
* Fix: Customer Chat widget now reflects the WordPress login state on init for logged-in users. v2.0.3 (#108) removed the JWT context-token fetch from `init()` to suppress a 401 console error for anonymous visitors. Side effect: `isAuthenticated()` (gated on the JWT) always returned false for logged-in users — so the menu showed "Log in" instead of "Log out", the escalation form rendered anonymous First name / Last name / Email fields, and server-side thread history (`loadServerThreads()`) was never called. Init now calls `getValidToken()` only when `config.isLoggedIn` is true, restoring pre-v2.0.3 behaviour for logged-in users while keeping the no-401 path for anonymous visitors.

= 2.1.7 - 2026/04/15 =
* Security: Per-IP rate limit on the anonymous Customer Chat escalation endpoint. Previously, `/wp-json/a3-ai/v1/customer-chat/send-escalation-email` was `permission_callback '__return_true'` with only a `wp_rest` nonce as the gate. A `wp_rest` nonce has a 12–24h lifetime and can be scraped from any Customer Chat page load, so a determined attacker could script POSTs and spam the partner escalation inbox at high volume. Flagged as HIGH severity in the Claude security review of PR #128 (v2.1.6) as a pre-existing issue from the v2.1.4 shipped code. Now rate-limited to 5 requests per 60 seconds per IP, returning HTTP 429 when exceeded.
* Implementation: Fixed-window counter backed by WordPress transients, keyed on `def_core_rl_` + md5 of the IP (so IPs aren't stored cleartext in `wp_options`). IP comes from `REMOTE_ADDR` only — `X-Forwarded-For` is deliberately not honoured because a client-supplied header cannot be trusted without a known-good proxy chain (it would let a spammer trivially rotate IPs). Sites behind a reverse proxy should rewrite `REMOTE_ADDR` server-side the usual way.
* Not rate-limiting the Setup Assistant escalation endpoint — it already requires the `def_admin_access` capability (v2.1.5), so it's only reachable by trusted admin users in the first place and spam isn't a meaningful threat model there.
* Note: Fixed-window counter is not atomic under high concurrency — two near-simultaneous requests can each see count=4 and each write count=5, letting 6 requests through in a window. Intentional: the goal is to blunt scripted spam, not to enforce a precise cap, and a small margin of error is acceptable for this threat model.

= 2.1.6 - 2026/04/15 =
* Fix: Customer Chat escalation — closing or cancelling the form now aborts an in-flight send. Previously `closeEscalation()` just hid the overlay and `trackAbort` / `untrackAbort` were pure bookkeeping helpers (they never called `controller.abort()`), so a mid-flight POST would complete in the background and a late success response could theoretically flip the hidden overlay back to its success state. Lifted the escalation fetch's `AbortController` to a module-scoped `escalationInflightController` so `closeEscalation()` can abort it on overlay close. The catch handler swallows `AbortError` silently so the user never sees an error state from an intentional cancel.
* Fix: Customer Chat escalation — `handleEscalationSubmit` now reads the response as text first and attempts JSON parse defensively. A non-JSON error body (HTML error page from the web server, an empty response, etc.) previously threw `Unexpected token <` out of the promise chain and dropped into the generic catch without any signal about what happened. Now the `{ ok, data }` shape still resolves with `data: null` in that case and the error state shows cleanly. Consistent with the equivalent Setup Assistant fix in v2.1.5.
* Note: both issues were flagged by Claude code review on PR #127 (Setup Assistant). They applied equivalently to the v2.1.4 Customer Chat code which was never reviewed. Fixing in a follow-up per @steve's scope preference (one channel per PR).

= 2.1.5 - 2026/04/15 =

Previously, asking Setup Assistant to escalate rendered an Accept/Decline card (v2.1.3) whose Accept button re-prompted the LLM in an infinite loop — no form, no send, no email. Replaced with a real inline form inside setup-assistant-drawer.js: Subject (pre-filled with a neutral default), Message textarea, Send / Cancel buttons, inline status line. On Send the browser POSTs `{subject, body}` directly to a new def-core route `POST /wp-json/a3-ai/v1/setup-assistant/send-escalation-email`, which validates the WP REST nonce, requires the caller to be a logged-in wp-admin user, reads display_name + user_email from `wp_get_current_user()` server-side (never client-supplied), prepends them to the body as a From/Site header, forces `channel=setup_assistant`, and delegates to the existing `send_escalation_email` handler. Recipients come from `def_core_escalation_setup_assistant` (the DEF partner email set in the admin Escalation tab). Conversation transcript is appended to the email body. Same architectural pattern as the v2.1.4 Customer Chat escalation fix — no cross-repo auth, no DEF involvement in the send path.
* New: `defSetupAssistant.escalationUrl` exposed via `wp_localize_script` so the drawer JS can call the new route.

* Security: `rest_setup_assistant_send` requires `def_admin_access` capability (not just a logged-in user). Flagged by Claude security review before merge.

= 2.1.4 - 2026/04/15 =
* Fix: Customer Chat escalation emails now send successfully. Previously, clicking Send on the escalation form always resulted in "Failed to send" for logged-in users on a3rev.com. Root cause: the send was routed browser → WP BFF proxy → DEF → def-core → wp_mail(), and the DEF → def-core hop used an auth model that broke after the BFF proxy migration (logged-in users carry no JWT but `is_authenticated=True` from trusted BFF headers, so neither the JWT nor the service-auth branch ran, and settings-fetch + send silently returned empty recipients). Rewritten to mirror the Staff AI Share pattern: browser builds the email body client-side and POSTs `{subject, body, reply_to?}` directly to a new def-core REST route `/wp-json/a3-ai/v1/customer-chat/send-escalation-email`, which whitelists the fields, forces `channel=customer` server-side, and delegates to the existing `send_escalation_email` handler (which reads recipients from `def_core_escalation_customer` and calls `wp_mail()`). Same pattern Staff AI Share already uses in production. Auth is WP REST nonce (same-origin), which works for both logged-in users and anonymous frontend visitors.
* New: `DEFCore` localized config now exposes `wpRestUrl` and `wpRestNonce` for same-origin REST calls from Customer Chat.
