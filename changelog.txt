*** Digital Employee Framework - Core Changelog ***

= 4.7.2 - 2026/06/11 =

= 4.7.0 - 2026/06/10 =
* Feature: Content Agent created posts now include real images. When you Generate a new post, the images the agent generates are sideloaded into your WordPress Media Library through a new hardened bridge (strict image-only validation on the actual bytes — PNG/JPEG/WebP only, never SVG or scripts, 10MB cap, safe filenames), the post gets a featured image, and the body's image spots become real Gutenberg image blocks with alt text. Your theme serves its own registered featured-image sizes automatically (WordPress generates them on upload from the high-res source), and every sideloaded image also gets an exact 1200×630 crop that's written to Yoast's Open Graph / Twitter share-image fields, so social shares look right everywhere. The "New post" review card now previews the featured image and inline images before you approve, and image cleanup is scoped so the agent can only ever remove images it created itself that no post is using — your own Media Library uploads are untouchable. Tenants without an image key keep the previous labelled placeholders — nothing breaks.
* Fix: after clicking Generate, the Content Drafts page now refreshes itself and shows the new draft card when it's ready (it previously said the draft "will appear shortly" but never refreshed — you had to reload the page).

= 4.3.2 - 2026/06/09 =
* Tweak: DEF backend traffic is now exempt from Wordfence throttling. When a bridge request passes HMAC signature verification, its egress IP is automatically added to the Wordfence allowlist — using Wordfence's own IP resolver and developer API, so it matches exactly what the firewall keys rate-limiting on. This keeps DEF's bursty authenticated audit/apply calls (which arrive from a large pool of rotating cloud egress IPs that a static allowlist can't track) from being rate-limited or blocked by the WAF. The allowlist is self-populating and de-duped — each IP is written to Wordfence config only the first time it's seen. It's a no-op on sites without Wordfence (no hard dependency), and the allowlisting is best-effort and isolated so it can never affect the authentication result. No pruning of rotated-out IPs (Wordfence exposes no clean removal API; the staleness risk is low and bounded by DEF's egress pool).

= 3.12.1 - 2026/05/28 =
* Fix: auto-sync incremental + delete-tracking exports returned zero rows on sites whose WordPress timezone is not UTC (e.g. Asia/Ho_Chi_Minh on a3rev). The Digital Employee backend sends the "modified after" watermark as ISO-8601 with an explicit timezone offset (e.g. `2026-05-28T10:00:13+00:00`). WordPress's own `WP_Date_Query::build_mysql_datetime()` parses that offset, then re-formats the value into the site's local timezone before building the SQL — but the column it compares against (`post_modified_gmt`) stays in UTC, so the comparison is silently off by the site's UTC offset and recently-edited content fails to match the filter. The export endpoints now normalize the watermark to a plain `Y-m-d H:i:s` UTC string before handing it to `date_query`, so the comparison is correct on any site timezone. Affects `/content/export`, `/products/export`, `/forums/export`, `/search/export`, and the trashed-posts query inside `/content/deleted` — all of them filter on `post_modified_gmt`.

= 3.6.2 - 2026/05/26 =
* Fix: products flagged "Exclude from Digital Employee knowledge" could still be added to cart. The customer chat resolves a product name → ID for add-to-cart through the live tool endpoint `/wp-json/a3-ai/v1/tools/wc/products` (`DEF_Core_Tools::wc_get_products_list()`), which queried `wc_get_products( status=publish, limit=-1 )` with NO exclusion filter — so the LLM matcher could still see and pick an excluded product's ID, even though exclusion correctly hid it from the Azure `search` index. Root cause: `_def_exclude_from_ingestion` was only enforced in the export/sync path, never in the live product tool. The endpoint now filters excluded products via `meta_query` plus an in-loop `get_post_meta()` guard (belt-and-suspenders, correct even if a WC version drops meta_query). Also: toggling the flag now busts the cached product list immediately (`DEF_Core_Cache::on_product_changed()` from the exclusion-change handler) — Bulk / Quick-Edit toggles update meta directly and don't fire `woocommerce_update_product`, which would otherwise leave the 1-hour cache stale. Staff AI's `wc/v3` REST passthrough is intentionally unchanged (internal staff may need full catalogue access); knowledge/chunk export endpoints are intentionally unchanged (the DEF backend reads the flag from those to deindex).

= 3.6.1 - 2026/05/25 =
* Fix: marking an item "Exclude from Digital Employee knowledge" now removes it from the live Digital Employee indexes on the next sync — this item only, no Full Sync needed. Previously the flag merely filtered the item out of future export pulls, so anything that had already been indexed before it was excluded lingered in DEF's `search` index indefinitely (the index population pass only upserts) and the chatbot kept surfacing it across recommend / add-to-cart / all tools. The flag change is now detected at the meta write itself — covering Gutenberg, classic editor, Quick Edit, bulk action, and programmatic updates — via `added/updated/deleted_post_meta` hooks on `_def_exclude_from_ingestion`. Two things happen per item: (1) the transition is recorded and surfaced in the `/wp-json/def-core/v1/content/deleted` feed under a new `excluded_ids` array (net-latest state wins, so a re-included item drops back out); (2) `post_modified` is bumped (direct SQL, no save_post/transition refire) so the incremental content + products export re-fetches the item — the chunk/knowledge index then deindexes the still-flagged item (via the existing `_def_exclude_from_ingestion` handling) or re-ingests it when re-included. Pairs with the DEF backend's new search-index delete pass (`/content/deleted` → `delete_objects`), which now drops `excluded_ids` (and permanent `deleted_ids`) from the `search` index. Admin meta-box + bulk-action copy updated: applied per item on the next sync, no Full Sync required.

= 3.5.0 - 2026/05/22 =
* Feature: Search-index export endpoint (`/wp-json/def-core/v1/search/export`, new `DEF_Core_Search_Export` class) for the DEF Search Tool — a metadata feed backing DEF's deterministic catalogue/site `search` index, separate from the content/chunk knowledge export. A `type` parameter selects the object family: a taxonomy (`product_cat` / `post_tag` / `product_brand` / `category` / …) returns term rows (categories/tags/brands as their own searchable result objects), while a post type (products / posts / pages / CPTs, or empty for all) returns item rows. Products additionally carry `sku`, rolled-up published `variation_skus`, `price` / `sale_price` / `on_sale` / `stock_status`, and `attributes` (name/value/slug) + concatenated `attributes_text`; taxonomy-term rows carry `object_count`. Paginated + incremental (`modified_after`), reusing the existing pull + `before_delete_post` delete-tracking + `_def_exclude_from_ingestion` exclusion — enforced at the source (published-only, non-password-protected, excluded items skipped). `tenant_id` is not emitted; the puller (DEFHO tenant platform) stamps it from the authenticated connection. `DEF_Core_Export::collect_taxonomy_terms()` was made public for reuse. This is the def-core half of the DEF Search Tool (pairs with DEF #306 export contract + #307 Azure index/sync); the DEFHO-side pull wiring and the `search` tool that queries the index follow.

= 3.4.1 - 2026/05/18 =
* Fix: page_context.current.page_id + page_context.current.product_id arrived as 0 / null in the chat-message request body on product pages (v3.4.0 regression caught in post-deploy smoke). Root cause: WP_Scripts::localize casts every top-level scalar in the localized array to a string before output — so the PHP-side `(int) get_queried_object_id()` arrived in JS as `"1234"` (string). The JS defensive `typeof === 'number'` check then rejected it and fell back to the default. Switched the payload injection from wp_localize_script to wp_add_inline_script + wp_json_encode (preserves types end-to-end). Added a JS-side `coerceInt()` helper for defense-in-depth — accepts both `number` and integer-string forms so any future serialization quirk also resolves cleanly. Affected fields restricted to top-level integer scalars; arrays / nested IDs (e.g. `terms[].term_id`) were unaffected because wp_localize_script doesn't recurse.

* Feature: Customer Chat now ships **page context** to DEF on every chat-message turn. = 2.1.7 - 2026/04/15 =
* Security: Per-IP rate limit on the anonymous Customer Chat escalation endpoint. Previously, `/wp-json/a3-ai/v1/customer-chat/send-escalation-email` was `permission_callback '__return_true'` with only a `wp_rest` nonce as the gate. A `wp_rest` nonce has a 12–24h lifetime and can be scraped from any Customer Chat page load, so a determined attacker could script POSTs and spam the partner escalation inbox at high volume. Flagged as HIGH severity in the Claude security review of PR #128 (v2.1.6) as a pre-existing issue from the v2.1.4 shipped code. Now rate-limited to 5 requests per 60 seconds per IP, returning HTTP 429 when exceeded.
* Implementation: Fixed-window counter backed by WordPress transients, keyed on `def_core_rl_` + md5 of the IP (so IPs aren't stored cleartext in `wp_options`). IP comes from `REMOTE_ADDR` only — `X-Forwarded-For` is deliberately not honoured because a client-supplied header cannot be trusted without a known-good proxy chain (it would let a spammer trivially rotate IPs). Sites behind a reverse proxy should rewrite `REMOTE_ADDR` server-side the usual way.
* Not rate-limiting the Setup Assistant escalation endpoint — it already requires the `def_admin_access` capability (v2.1.5), so it's only reachable by trusted admin users in the first place and spam isn't a meaningful threat model there.
* Note: Fixed-window counter is not atomic under high concurrency — two near-simultaneous requests can each see count=4 and each write count=5, letting 6 requests through in a window. Intentional: the goal is to blunt scripted spam, not to enforce a precise cap, and a small margin of error is acceptable for this threat model.

Root cause: the send was routed browser → WP BFF proxy → DEF → def-core → wp_mail(), and the DEF → def-core hop used an auth model that broke after the BFF proxy migration (logged-in users carry no JWT but `is_authenticated=True` from trusted BFF headers, so neither the JWT nor the service-auth branch ran, and settings-fetch + send silently returned empty recipients). Same pattern Staff AI Share already uses in production. Auth is WP REST nonce (same-origin), which works for both logged-in users and anonymous frontend visitors.
* New: `DEFCore` localized config now exposes `wpRestUrl` and `wpRestNonce` for same-origin REST calls from Customer Chat.
